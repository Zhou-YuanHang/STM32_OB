# 中断向量表（Interrupt Vector Table）

## 一句话定义

> **中断向量表就是一张存放在 Flash 起始地址（0x00000000）的"跳转地址表"，硬件根据异常编号自动查表，找到对应的中断服务函数入口地址并跳转过去。**

---

## 一、为什么叫"向量"？

"向量"（Vector）在这里不是数学里的矢量，而是**"指向目标的指针/地址"**的意思。

| 术语 | 含义 |
|------|------|
| **中断向量** | 一个 32 位的地址值，指向对应的中断处理函数 |
| **中断向量表** | 所有这些向量按异常编号顺序排成的表 |

---

## 二、向量表在内存中的位置

STM32F103 上电后，Flash 被映射到 `0x00000000`，向量表就放在 Flash 的开头：

```
Flash 起始地址（0x08000000）            SRAM 起始（0x20000000）
    │                                        │
    ▼                                        ▼
┌─────────────────┐                   ┌─────────────┐
│   向量表         │ ← 同时也映射到   │             │
│   (Vector Table) │   0x00000000     │   运行数据   │
│                 │                  │   (.data     │
│   用户代码       │                  │    .bss      │
│   (.text)       │                  │    栈 Stack)  │
└─────────────────┘                   └─────────────┘
```

---

## 三、向量表的结构（一张表格看懂）

以 STM32F103 为例，向量表的前几项是 **系统异常**，后面是 **外设中断**：

| 位置偏移 | 向量名称 | 异常号 | 作用 |
|----------|---------|--------|------|
| `0x0000_0000` | `__initial_sp` | — | **不是地址，是栈顶值**！芯片一上电把这个值装入 SP |
| `0x0000_0004` | `Reset_Handler` | 1 | 复位向量，芯片上电后 PC 指针跳到这里 |
| `0x0000_0008` | `NMI_Handler` | 2 | 不可屏蔽中断（硬件故障） |
| `0x0000_000C` | `HardFault_Handler` | 3 | ⭐ 硬错误——参数出错率最高的异常 |
| `0x0000_0010` | `MemManage_Handler` | 4 | MPU 内存保护错误 |
| `0x0000_0014` | `BusFault_Handler` | 5 | 总线错误（非法地址访问） |
| `0x0000_0018` | `UsageFault_Handler` | 6 | 用法错误（除零、未定义指令等） |
| `0x0000_003C` | `SysTick_Handler` | 15 | 系统滴答定时器中断 |
| `0x0000_0058` | `EXTI0_IRQHandler` | 22 | 外部中断线 0 |
| `0x0000_00D4` | `USART1_IRQHandler` | 53 | 串口 1 中断 |

> 规律：向量地址 = `0x00000000` + 异常号 × 4（每个向量占 4 字节）

---

## 四、上电/中断时硬件怎么查表？

### 场景 1 —— 芯片上电复位

```
① 硬件自动从 0x00000000 读 4 字节 → 装入 SP（栈指针）
② 硬件自动从 0x00000004 读 4 字节 → 装入 PC（程序计数器）
③ CPU 开始执行 Reset_Handler 里的代码
```

### 场景 2 —— 串口收到数据，触发 USART1 中断

```
① 硬件自动找到 USART1 的异常号 = 53
② 计算向量地址 = 0x00000000 + 53 × 4 = 0x000000D4
③ 从 0x000000D4 读 4 字节 → 这就是 USART1_IRQHandler 的函数地址
④ CPU 跳转到该地址，执行中断服务函数
⑤ 中断函数执行完 → 返回主程序（硬件自动恢复现场）
```

## 五、在启动文件中的实际样子

```asm
; startup_stm32f10x_md.s 中的向量表定义

__Vectors       DCD     __initial_sp             ; 0x0000_0000  栈顶
                DCD     Reset_Handler            ; 0x0000_0004  复位
                DCD     NMI_Handler              ; 0x0000_0008  NMI
                DCD     HardFault_Handler        ; 0x0000_000C  HardFault
                DCD     MemManage_Handler        ; 0x0000_0010  MPU
                DCD     BusFault_Handler         ; 0x0000_0014  总线错误
                DCD     UsageFault_Handler       ; 0x0000_0018  用法错误
                ; ... 后面是外设中断向量
                DCD     USART1_IRQHandler        ; USART1 中断
                DCD     USART2_IRQHandler        ; USART2 中断
                ; ...
```

`DCD`（Define Constant Data）是 ARM 汇编伪指令，意思是**在当前位置放一个 32 位常量**——这里的"常量"就是各函数的地址。

---

## 六、为什么向量表必须在 Flash 开头？

因为 Cortex-M3 的硬件设计就是上电后**固定从 `0x00000000` 取前两个 32 位值**。硬件没有"找一下向量表在哪"这个概念——它只认地址 `0x00000000`。

```
Cortex-M3 上电行为（硬件写死，不可改）：

    SP = *(volatile uint32_t *)0x00000000;
    PC = *(volatile uint32_t *)0x00000004;
```

这点和 PC 的 x86 完全不同。x86 上电后跳转到固定的 BIOS 地址（`0xFFFFFFF0`），而 ARM Cortex-M **通过向量表实现间接跳转**，更灵活。

## 七、相关概念辨析

| 容易混淆的概念 | 区别 |
|---------------|------|
| **中断向量表 vs 外设寄存器表** | 向量表放的是函数的地址（跳转目标）；寄存器表放的是外设的控制值（配置参数） |
| **中断向量 vs 中断服务函数** | 向量是指向函数的指针（4 字节地址）；服务函数是真正干活的代码 |
| **向量表在 Flash vs 在 SRAM** | 默认在 Flash；可以设置 `VTOR` 寄存器重映射到 SRAM（做动态向量表/IAP 升级时用） |
| **异常 vs 中断** | ARM 把编号 1~15 的叫异常（系统级），16 以上的叫中断（外设级）。本质一样，都查向量表 |

---

## 八、一句话总结

```
中断发生后：

    硬件干了三件事：
    ① 保存当前现场（xPSR、PC、LR、R0-R3、R12 入栈）
    ② 根据异常号查向量表 → 找到中断服务函数地址
    ③ 跳转到该地址执行

    程序员要干的事：
    ① 在启动文件中把向量表写对
    ② 在 stm32f10x_it.c（或任意 .c 文件）中实现同名函数
    ③ 函数名必须和向量表中的名字完全一致！
```
